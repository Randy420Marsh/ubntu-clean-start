Ubuntu 22.04 to 24.04 Failed Release Upgrade Recovery Manual

This guide walks you through recovering from a failed do-release-upgrade that left you with no network, broken packages, double GNOME panels, or a half-installed NVIDIA / CUDA stack — and ends with a clean, stable Ubuntu 24.04 desktop or headless server.

Before you start. Every step assumes you have a current backup of /home, /etc, and anything else you care about. Several sections delete files. Don't run them blind.

1. Boot a Ubuntu 24.04 Live USB and chroot

  1. Download the official Ubuntu 24.04 Desktop ISO and write it to a USB (Rufus, balenaEtcher, dd, etc.).
  2. Boot the USB and choose Try Ubuntu.
  3. Open Terminal and identify your installed root partition: 
    lsblk -f
    Look for the largest ext4 partition that has your existing system. If you have a separate /boot or /boot/efi, note them too.
  4. Mount the root partition. Replace sdXY with your actual device (e.g. nvme0n1p2): 
    sudo mount /dev/sdXY /mnt
  5. If you have a separate /boot and / or an EFI System Partition, mount them inside the new root so that grub-install and update-initramfs can do their job: 
    sudo mount /dev/sdXZ /mnt/boot           # only if separate /boot exists
sudo mount /dev/sdXW /mnt/boot/efi       # only if EFI partition exists
  6. Bind-mount the kernel virtual filesystems, copy DNS, and chroot in: 
    for d in dev dev/pts proc sys run; do
    sudo mount --bind /$d /mnt/$d
done
sudo cp /etc/resolv.conf /mnt/etc/resolv.conf
sudo chroot /mnt /bin/bash
LVM, LUKS, or Btrfs root? The plain mount /dev/sdXY /mnt recipe will not work. You need to unlock the LUKS container first (cryptsetup luksOpen ...), activate the LVM volume group (vgchange -ay), or mount the correct Btrfs subvolume (mount -o subvol=@ /dev/... /mnt). If you are not sure, ask before continuing.

2. Fix broken packages inside the chroot

Run these in order, inside the chroot:

dpkg --configure -a
apt --fix-broken install -y
About removing third-party DKMS drivers. A widely-shared recipe for fixing failed Jammy upgrades tells you to purge v4l2loopback-dkms, rtl8812au-dkms, 8188eu-dkms, and ax88179a-dkms outright. Only do that if you know you do not depend on them. Specifically:
  • rtl8812au-dkms drives many USB WiFi adapters (Alfa, TP-Link Archer T-series, etc.). Removing it can leave your laptop with no WiFi after reboot.
  • ax88179a-dkms drives ASIX AX88179 / AX88179A USB 3 Gigabit ethernet adapters.
  • 8188eu-dkms drives RTL8188EU-based USB WiFi.
  • v4l2loopback-dkms is needed by OBS Virtual Camera, Snap Camera, etc.
Check first with lsmod | grep -E 'rtl|ax88|8188|v4l2loopback' and lsusb. Only purge the ones you do not use.

Then, only for modules you do not need, purge them one at a time:

# Example — adapt to your hardware:
apt purge -y v4l2loopback-dkms
apt purge -y 8188eu-dkms
apt --fix-broken install -y

If you hit a libslang2-dev overwrite error (a known Jammy to Noble upgrade bug), force the overwrite:

dpkg -i --force-overwrite /var/cache/apt/archives/libslang2-dev_*.deb
apt --fix-broken install -y

3. Clean APT sources (jammy to noble)

Do not restore /etc/apt/sources.list.distUpgrade on top of /etc/apt/sources.list. That file is a backup of the pre-upgrade jammy sources that do-release-upgrade saves so you can roll back. Copying it over the live sources will pin you back to jammy, which is the opposite of what we want.

On Ubuntu 24.04 the primary sources file is the deb822-format /etc/apt/sources.list.d/ubuntu.sources, not the legacy /etc/apt/sources.list. Handle both:

# Back up everything first so you can always revert.
mkdir -p /root/apt-backup-$(date +%Y%m%d)
cp -a /etc/apt/sources.list /etc/apt/sources.list.d /root/apt-backup-$(date +%Y%m%d)/ 2>/dev/null || true

# If you have the deb822 file, make sure it points at noble.
if [ -f /etc/apt/sources.list.d/ubuntu.sources ]; then
    sed -i 's/\bjammy\b/noble/g' /etc/apt/sources.list.d/ubuntu.sources
fi

# Rewrite any remaining legacy lists.
if [ -f /etc/apt/sources.list ]; then
    sed -i 's/\bjammy\b/noble/g' /etc/apt/sources.list
fi

# Third-party PPAs and .sources files: only rewrite jammy to noble if the
# upstream actually has a noble repo. If you are not sure, comment the line
# out instead (prefix with #) and re-enable it later.
shopt -s nullglob
for f in /etc/apt/sources.list.d/*.list /etc/apt/sources.list.d/*.sources; do
    [ "$f" = "/etc/apt/sources.list.d/ubuntu.sources" ] && continue
    echo "Review: $f"
    grep -nH 'jammy' "$f" || true
done
shopt -u nullglob

Once the sources are sane, refresh and upgrade:

rm -rf /var/lib/apt/lists/*
apt clean
apt update
apt full-upgrade -y
apt autoremove -y
apt --fix-broken install -y

4. Leave the chroot, unmount, reboot, verify

exit                              # leave the chroot
sudo umount -R /mnt               # recursive — handles /dev/pts, /boot/efi, etc.
sudo reboot

If umount -R complains about busy mounts, find what is using them:

sudo lsof +D /mnt 2>/dev/null
sudo fuser -vm /mnt

After reboot, check:

lsb_release -a
systemctl status NetworkManager      # or systemd-networkd
ping -c 3 1.1.1.1

5. Fix the GNOME desktop (double panel, extensions)

If you have a double bottom panel, ghost docks, or generally broken extensions, reset the user-level state:

# Remove user-installed extensions (system extensions in /usr/share are left alone).
rm -rf ~/.local/share/gnome-shell/extensions/*
rm -rf ~/.cache/gnome-shell

# Disable everything that is currently enabled, then let GNOME re-enable
# the defaults on next login.
gsettings set org.gnome.shell enabled-extensions "[]"
gsettings set org.gnome.shell disabled-extensions "[]"

# Wipe dconf state for extensions.
dconf reset -f /org/gnome/shell/extensions/
gnome-extensions reset is not a real subcommand. Older guides recommend it; it just prints command not found for reset. Use the gsettings / dconf commands above instead.

Now log out and log back in. On Wayland sessions, do not run systemctl restart gdm — that will kill your session abruptly and can leave your user bus in a weird state. Use the regular Log Out option, or from a TTY:

loginctl terminate-user "$USER"

After logging back in you should have a clean default GNOME shell with only the top bar and the standard Ubuntu dock.

6. Purge old NVIDIA / CUDA / cuDNN safely

This repository ships a hardened purge script, safe_clean_nvidia.sh, designed to be safe on a headless server over SSH. It will:

  • stop any active display manager (gdm3 / sddm / lightdm), if one exists;
  • unload nvidia, nvidia_drm, nvidia_modeset, nvidia_uvm, and nvidia_peermem before purge;
  • purge every nvidia-*, libnvidia-*, cuda-*, libcudnn*, libcublas*, libcufft*, libcurand*, libcusolver*, libcusparse*, libnpp*, libnccl*, libnvjpeg*, libnvinfer*, libcutensor*, tensorrt*, and xserver-xorg-video-nvidia* package;
  • delete CUDA / NVIDIA apt sources, signing keys, residual /usr/local/cuda*, /usr/src/nvidia-*, /etc/ld.so.conf.d/cuda*.conf, /etc/profile.d/cuda*.sh, /etc/modprobe.d/nvidia*.conf, and /etc/OpenCL/vendors/nvidia.icd;
  • remove only the rc-state packages that are clearly NVIDIA / CUDA related (it does not blanket-purge every rc package on the system);
  • back up /etc/X11/xorg.conf if it references the nvidia driver, rather than deleting it outright;
  • refresh ldconfig at the end.

Run it with --dry-run first to see exactly what it will touch:

sudo ./safe_clean_nvidia.sh --dry-run

Then run for real:

sudo ./safe_clean_nvidia.sh

Optional flags:

  • --restore-nouveau — also remove /etc/modprobe.d/blacklist-nouveau.conf and rebuild initramfs so the open-source nouveau driver loads again on reboot.
  • --purge-extra-dkms — also remove v4l2loopback-dkms, rtl8812au-dkms, 8188eu-dkms, and ax88179a-dkms. Only use this if you have confirmed you do not depend on those drivers (see section 2).
  • -y / --yes — non-interactive; skip the reboot prompt and exit. Useful for SSH automation.
The script never silently removes WiFi / USB-ethernet / virtual-camera drivers, never blanket-purges all rc packages, and never deletes blacklist-nouveau.conf unless you explicitly ask for it. These are the three real footguns in most public NVIDIA cleanup scripts.

7. Boot to terminal-only mode (TTY / GRUB / target 3)

Several of the steps below — most importantly installing the NVIDIA driver in section 9 — need the display server to be stopped. "Boot to terminal" is the modern equivalent of old SysV runlevel 3; under systemd it is multi-user.target. Three ways to get there, pick whichever fits the situation:

Option A — One-shot, edit the entry at the GRUB menu

  1. Reboot. If you don't see the GRUB menu, hold Shift (BIOS boot) or tap Esc (UEFI boot) during firmware POST.
  2. Highlight the Ubuntu entry, press e to edit it.
  3. Find the line that starts with linux (it ends with something like ro quiet splash $vt_handoff).
  4. Append a single 3 at the end of that line: 
    linux /boot/vmlinuz-... ro quiet splash $vt_handoff 3
    That tells systemd to start in multi-user.target. If 3 is ignored on your distro for any reason, use the explicit form instead: 
    linux /boot/vmlinuz-... ro quiet splash $vt_handoff systemd.unit=multi-user.target
  5. Press Ctrl-X (or F10) to boot. This boot only comes up in a TTY. The next normal boot will go back to your usual default target.

Option B — Permanent: make the TTY the default target

sudo systemctl set-default multi-user.target
sudo reboot

To go back to a graphical login screen:

sudo systemctl set-default graphical.target
sudo reboot

You can check what is set right now with:

systemctl get-default

Option C — Permanent via /etc/default/grub

Useful if you want the kernel command line to be persistent (for example, also passing nvidia-drm.modeset=1 at the same time). Edit /etc/default/grub and find GRUB_CMDLINE_LINUX_DEFAULT:

sudo nano /etc/default/grub
# change e.g. from:
#   GRUB_CMDLINE_LINUX_DEFAULT="quiet splash"
# to:
#   GRUB_CMDLINE_LINUX_DEFAULT="quiet splash systemd.unit=multi-user.target"
sudo update-grub
sudo reboot

To undo, remove the systemd.unit=... token and sudo update-grub again.

Option D — Drop to TTY in the running session (no reboot)

sudo systemctl isolate multi-user.target     # stops X / Wayland *now*
sudo systemctl isolate graphical.target      # brings the desktop back
Option D kills your graphical session immediately. Save your work first. If you are connected over SSH this is fine — your SSH session lives in multi-user.target and will survive.
Inside a TTY you log in with your normal username and password. Switch between virtual terminals with Ctrl-Alt-F2 through Ctrl-Alt-F6. The desktop, when it is running, usually lives on Ctrl-Alt-F1 or F7.

8. Disable the kernel microcode loader (limited-BIOS-ROM boards, e.g. ASRock Z170 PG)

Some older LGA1151 / Skylake-era motherboards — the ASRock Z170 PG and other Z170 boards in particular — ship with a BIOS flash chip too small to hold the full set of Intel microcode revisions plus the rest of the firmware. The vendor's BIOS has either an old microcode or no microcode at all, and Linux is expected to apply a current revision early at boot from an initrd blob.

On those boards, the in-kernel microcode loader can do one of three annoying things:

  • hang the box during early boot (you see the EFI stub messages, then a black screen forever);
  • boot but with unstable CPU behaviour — random freezes, MCE events in dmesg, USB / SATA controller drops;
  • force a Spectre / L1TF / MDS mitigation set that the user wants to opt out of for performance reasons.

The fix is to stop Linux from loading microcode at boot and let the BIOS-shipped revision stay in effect. Three ways to do it, pick one.

Security tradeoff. Microcode updates carry fixes for Spectre, Meltdown, L1TF, MDS, Downfall, INCEPTION, etc. If you disable the kernel microcode loader and your BIOS does not ship a recent microcode, you lose those mitigations. Only do this if you have actually verified that the kernel microcode loader is the cause of the problem you are seeing (e.g. with git bisect-style testing of the intel-microcode package versions). On a network-facing machine, prefer fixing the BIOS instead.

Option A — kernel command line, dis_ucode_ldr

This is the per-boot kernel switch. Edit /etc/default/grub and append dis_ucode_ldr to GRUB_CMDLINE_LINUX_DEFAULT:

sudo sed -i 's/^GRUB_CMDLINE_LINUX_DEFAULT="\(.*\)"$/GRUB_CMDLINE_LINUX_DEFAULT="\1 dis_ucode_ldr"/' /etc/default/grub
sudo update-grub
sudo reboot

Verify after reboot:

# Should print: "dis_ucode_ldr ..."
cat /proc/cmdline | tr ' ' '\n' | grep ucode

# Should print something like:
#   microcode: Microcode loader disabled by kernel cmdline
dmesg | grep -iE 'microcode|ucode' | head

To undo: remove the token from GRUB_CMDLINE_LINUX_DEFAULT and sudo update-grub.

Option B — remove the intel-microcode package

This stops the package from ever building an early-initrd microcode blob in the first place. Useful when Option A is not enough because a service (e.g. iucode-tool) is re-applying microcode later in boot.

sudo apt purge intel-microcode iucode-tool
sudo update-initramfs -u
sudo reboot

To undo: sudo apt install intel-microcode and reboot.

Option C — pin a known-good microcode version

If a specific recent intel-microcode revision is the problem but you do still want microcode (the common Z170 case is that one specific upload broke things; the previous one was fine), pin and hold the older version:

apt list --all-versions intel-microcode
sudo apt install intel-microcode=<older-version>
sudo apt-mark hold intel-microcode
sudo update-initramfs -u
sudo reboot

To undo: sudo apt-mark unhold intel-microcode && sudo apt install intel-microcode.

Verifying the running revision

# What microcode revision the running CPU has loaded:
grep -m1 microcode /proc/cpuinfo
# Decoded:
sudo dmidecode -t processor | grep -i 'version\|signature'
# What Linux tried to apply (if anything):
dmesg | grep -iE 'microcode|ucode'
On ASRock Z170 PG specifically: the last BIOS that fits cleanly is 7.40 (LGA1151-v1 line). The intel-microcode package revisions that have been reported to cause boot hangs on this board cluster around the "Plundervolt" / "CrossTalk" Skylake rollouts (3.20191112.1 and later). If you have to keep microcode for security reasons, pinning 3.20191115.2ubuntu0.20.04.1 or an earlier stable build (Option C) is usually the smallest regression. Always check Intel's public microcode-data-files repo for any vendor notices affecting your specific stepping (cat /proc/cpuinfo | grep -m1 -E 'model|stepping').

Verifying microcode is no longer loaded at boot

After purging intel-microcode (Option B) or adding dis_ucode_ldr (Option A), microcode can still be applied at boot from any of these six places. Run the checks below; they confirm whether the kernel is still loading a blob.

Important context: the BIOS itself always applies the microcode it was shipped with at power-on, regardless of OS — CPU microcode is volatile and has to be reloaded every cold boot. The OS-side loader's only job is to override the BIOS-baked microcode with a newer revision after boot. "Disabling microcode" therefore means "do not let the OS override the BIOS revision", not "run the CPU with no microcode at all" (which is not a thing).

# 1. What the running kernel actually did this boot. Look for
#    "microcode updated early to revision 0x..." or "microcode patch
#    loaded". If you see neither, the kernel did not override the
#    BIOS-baked microcode. If you see "early microcode loader: lookup
#    failed", that is also fine - it tried, found nothing, gave up.
sudo dmesg | grep -iE 'microcode|ucode'

# 2. Current revision running on the CPU. Compare with the "CPU
#    Microcode Revision" / "Microcode Patch ID" line in BIOS Setup.
grep -i microcode /proc/cpuinfo | head -1
cat /sys/devices/system/cpu/microcode/version 2>/dev/null

# 3. Kernel command line as actually applied. If you used Option A
#    you must see "dis_ucode_ldr" here. If not, /etc/default/grub was
#    not saved, or update-grub was not run, or you booted an old
#    GRUB entry.
cat /proc/cmdline

# 4. Microcode firmware files still on disk?
# Ubuntu 24.04 uses usrmerge, so /lib/firmware is a symlink to
# /usr/lib/firmware. Both invocations look at the same directory; either
# path may show up in plocate output.
ls -la /usr/lib/firmware/intel-ucode/ 2>/dev/null
ls -la /usr/lib/firmware/amd-ucode/   2>/dev/null
ls -la /lib/firmware/intel-ucode/     2>/dev/null
ls -la /lib/firmware/amd-ucode/       2>/dev/null
sudo find / -xdev -type f \( -name '*ucode*' -o -name '*microcode*' \) \
    2>/dev/null | grep -v -E '/(proc|sys|run)/'

# 5. GRUB-style early microcode image in /boot?
ls -la /boot/intel-ucode.img /boot/amd-ucode.img 2>/dev/null

# 6. Initramfs for the running kernel still bundling microcode?
sudo lsinitramfs /boot/initrd.img-$(uname -r) 2>/dev/null \
    | grep -iE 'ucode|microcode'

# 7. Does grub.cfg still chainload a microcode image?
sudo grep -niE 'ucode|microcode' /boot/grub/grub.cfg \
    /etc/grub.d/*microcode* 2>/dev/null

# 8. Any microcode-related packages or services still installed?
dpkg -l | grep -iE 'microcode|ucode|iucode'
systemctl list-unit-files | grep -iE 'microcode|ucode'

How to clean up each remaining loader:

  • Check #4 lists files in /usr/lib/firmware/intel-ucode/ (or the usrmerge symlink /lib/firmware/intel-ucode/): sudo rm -rf /usr/lib/firmware/intel-ucode is safe after the package is purged. With dis_ucode_ldr set the kernel will not load files from this directory anyway, so this is a tidiness cleanup, not a functional one.
  • Check #5 returns /boot/intel-ucode.img: 
    sudo rm -f /boot/intel-ucode.img
sudo update-grub
  • Check #6 lists kernel/x86/microcode/GenuineIntel.bin: initramfs is still bundling microcode. Force a rebuild: 
    sudo update-initramfs -u -k all
  • Check #7 returns matches in /boot/grub/grub.cfg: after the above two cleanups, regenerate the GRUB config: 
    sudo update-grub
  • Check #8 lists anything other than iucode-tool (which is just the tool, not firmware): 
    sudo apt purge intel-microcode amd64-microcode iucode-tool
  • Check #3 is missing dis_ucode_ldr: redo Option A — edit /etc/default/grub, sudo update-grub, reboot.

Once all eight checks come back clean, the CPU is running on whatever microcode revision the BIOS shipped with — verified in #2 by matching it against the BIOS Setup "CPU Microcode Revision" value. From the BIOS-vendor perspective this is exactly the pre-Plundervolt configuration the Z170 PG was tested against.

Expected-but-alarming-looking dmesg lines

Two messages that show up after disabling the loader are expected and not bugs:

  • Unknown kernel command line parameters "text dis_ucode_ldr 3", will be passed to user space. This is the kernel's core parameter parser complaining that it does not own those three tokens. dis_ucode_ldr is registered as an early_param, parsed before the core parser runs; text and 3 are systemd shorthand for "console-only, multi-user.target". All three are handled correctly by the right consumer; the "unknown" message is the core parser saying "I do not know these, here you go userspace," which is fine. Confirm the dis_ucode_ldr half actually fired by looking for a later dis_ucode_ldr line in dmesg (it shows up around [ 0.6xx ]).
  • MDS: Vulnerable: Clear CPU buffers attempted, no microcode, and equivalent lines for TAA, MMIO Stale Data, SRBDS, GDS. These say "the CPU is still architecturally vulnerable to these because the microcode-based mitigation is not loaded — falling back to buffer-clearing if we can." That is the expected consequence of running without an OS-side microcode update on a 6th-gen Skylake. Not a regression; you accepted this tradeoff by choosing the "limited BIOS ROM, can't fit current microcode" path. If you ever want the microcode-based mitigations back, you have to either re-enable the OS-side loader (Option A, undo) or update the BIOS to a revision that includes a newer microcode (only an option if a newer BIOS exists for your board and it still fits the ROM).

9. Reinstall the NVIDIA driver (.run + MOK + Secure Boot)

This repository ships driver-install.sh, which wraps the upstream NVIDIA .run installer with:

  • auto-detection of the .run file in the current directory (no hard-coded version);
  • generation of a Machine Owner Key (MOK) if you don't already have one, plus an enrolment walk-through via mokutil --import when Secure Boot is on;
  • GPU-generation detection: it will refuse to pass --kernel-module-type=open on Pascal / Maxwell / Kepler hardware (where the open modules do not work) and fall back to the closed modules automatically;
  • a guided step that walks you through booting to a TTY (section 7) before swapping out the in-use kernel modules.

Typical run:

# Download the .run installer from
#   https://www.nvidia.com/Download/index.aspx
# into the current directory, then:
sudo ./driver-install.sh

Useful flags:

  • --setup-mok — only generate the MOK keypair and kick off enrolment, then exit. Reboot, complete enrolment in the blue MokManager screen, then rerun the script for real.
  • --open / --proprietary — force the kernel-module flavour.
  • --skip-isolate — don't offer to switch to multi-user.target (assumes you already did).
  • /path/to/NVIDIA.run — pass the installer explicitly if you have several .run files in the directory.
If Secure Boot is on, do not skip MOK enrolment. A signed module whose certificate is not in shim's MOK list will be silently rejected at modprobe time and nvidia-smi will tell you the driver is not loaded. Verify with: 
sudo mokutil --sb-state          # "SecureBoot enabled"?
sudo mokutil --list-enrolled | grep -i 'Local NVIDIA'

10. Troubleshooting: "MOK storage full" / EFI NVRAM exhausted

Symptom: at the blue MokManager screen after mokutil --import, the firmware prints "MOK storage full" (or "failed to allocate variable") and refuses to enrol the key. After reboot the kernel modules signed with that MOK still fail to load.

What this actually means: not that you have too many MOKs. It means the EFI variable region in firmware NVRAM is exhausted. Mainboards have a fixed-size NVRAM area (often only ~64 KB on Skylake-era ASRock boards) that is shared by:

  • Secure Boot key databases (db, dbx, KEK, PK);
  • MOK variables (MokListRT, MokListXRT, MokListTrustedRT, and the pending-request siblings MokNew, MokDel, MokAuth);
  • UEFI boot entries (Boot####, BootOrder, BootCurrent);
  • Linux pstore crash dumps written through the efivars backend (dump-type0-*-#-#-#); these accumulate every time the kernel panics or you call echo c > /proc/sysrq-trigger;
  • Firmware-side data (HII forms, certificate revocation lists, CSM compatibility tables on hybrid BIOSes, OEM logos, anti-rollback counters, fTPM state on boards with firmware-TPM, ASRock-specific "Instant Flash" / "Easy Driver Installer" state).

Clearing just the MOK keys from the BIOS does not reclaim the space taken by everything else. You have to find what is actually consuming the region.

Read this whole section before touching anything in /sys/firmware/efi/efivars/. A wrong rm in that directory can brick the board (historically, rm -rf on efivars nuked some Samsung laptops). Recent kernels make the dangerous variables immutable with chattr +i, but treat the directory as live firmware state, not a filesystem.

Step 1 — see what is actually consuming NVRAM

# Top consumers of EFI variable storage, largest first:
sudo ls -laS /sys/firmware/efi/efivars/ | head -30

# Total in-use vs total available (kernel reports both via dmesg):
sudo dmesg | grep -iE 'efivars|efi: |variable storage|nvram' | head

# A nicer breakdown if you have the 'efivar' tool:
sudo apt install -y efivar
sudo efivar -l | wc -l
sudo efivar -l | head -40

The five things that most commonly explain a "full" NVRAM on a long-lived Linux install are, roughly in order of how often they are actually the culprit:

  1. An oversized dbx (Secure Boot revocation list) that has been grown by fwupd applying the periodic LVFS dbx update. The factory dbxDefault is typically ~3–4 KB; the current Microsoft-published revocation list (BlackLotus, CVE-2023-24932, etc.) is ~17–30 KB. On Skylake-era ASRock boards (Z170 PG, H170, B150) this single variable can consume a quarter to a third of the whole NVRAM region;
  2. dozens of dump-type0-* entries from kernel panics (these can each be several KB);
  3. dozens of stale Boot#### entries from old kernels / live USBs / Windows installs that were never cleaned up;
  4. a wedged pending MOK request (MokNew, MokDel, MokAuth) left after a failed previous import;
  5. the actual MokListRT if you have legitimately enrolled many keys over time.

Step 1.5 — handling an oversized dbx (the most common Z170 PG cause)

If your ls -laS shows dbx-* as the single largest variable (typically 17–30 KB), the UEFI revocation list itself is what is filling NVRAM. This is not a Linux bug — the list grows every time fwupd-refresh.timer applies a new revocation set from LVFS. Microsoft has expanded it significantly since 2022 to revoke BlackLotus-affected bootloaders. Reference: on a Z170 PG you should see something like:

-rw-------  17840  ...  dbx-d719b2cb-3d3a-4596-a3bc-dad00e67656f
-rw-r--r--   3728  ...  dbxDefault-8be4df61-93ca-11d2-aa0d-00e098032b8c

The 17 840 B runtime dbx on a board with ~64 KB of NVRAM is already eating ~28 % of the region by itself. Combined with ASRock-internal StdDefaults / VARSTORE_OCMR_SETTINGS_NAME (~14 KB + ~9 KB on Z170 PG) and the Secure Boot db / KEK / PK variables (~17 KB), there is no room left for MokManager to allocate MokNew.

Fix, in order:

  1. Stop fwupd from growing dbx further (do this before any reset, so it cannot re-apply the update at the next boot):

    sudo systemctl disable --now fwupd-refresh.timer
sudo systemctl mask fwupd-refresh.service
# Optional, if you do not use fwupd at all for firmware updates:
sudo apt purge fwupd fwupd-signed

  2. Reset the runtime dbx to the factory default from BIOS Setup. On ASRock 100/200-series:

    • Boot to BIOS Setup → Security → Secure Boot → Key Management.
    • Select Forbidden Signatures Database (DBX) → Delete (or Reset to Setup Mode followed by Restore Factory Keys).
    • Save and exit.
    • Verify on the booted system: 
      sudo ls -laS /sys/firmware/efi/efivars/dbx-*
      should now be ~3.7 KB, matching dbxDefault.

  3. Security tradeoff: by reverting dbx you lose the protection against bootloaders that Microsoft has revoked since the BIOS was manufactured (most importantly BlackLotus and the CVE-2023-24932 cluster). This is an acceptable tradeoff if you:

    • do not dual-boot a current Windows install that relies on the dbx update for Defender;
    • are using your own MOK to sign your own modules (the whole reason this section exists);
    • and you understand that a malicious bootloader that was on the dbx will once again be allowed to run if it ever ends up on your ESP.

Step 2 — clear Linux pstore crash dumps

These are the most common culprit and the safest to remove. pstore is the kernel's crash-dump backend; on UEFI systems it routes through EFI variables by default. Each panic leaves a 1–4 KB blob in NVRAM that nobody ever cleans up.

# See what is there:
sudo ls -lh /sys/fs/pstore/ 2>/dev/null

# Delete them (this also removes the matching efivars entries):
sudo rm -f /sys/fs/pstore/*

# Stop them from coming back: log to disk instead of efivars.
sudo tee /etc/modprobe.d/pstore.conf >/dev/null <<'EOF'
# Prefer a filesystem-backed pstore so crash dumps do not fill EFI NVRAM.
options efi_pstore pstore_disable=1
EOF
sudo update-initramfs -u

For a stronger fix on boards with chronically small NVRAM, add efi_pstore.pstore_disable=1 to GRUB_CMDLINE_LINUX_DEFAULT in /etc/default/grub and sudo update-grub. Crashes still get logged via kdump / journald, just not into EFI NVRAM.

Step 3 — prune stale UEFI boot entries

On a system that has gone through several reinstalls (Ubuntu 24.04, the prior 22.04, a Windows partition, a live USB, etc.), efibootmgr -v will often show 10–20 Boot#### entries pointing at filesystems that no longer exist.

# List current entries with their targets:
sudo efibootmgr -v

# Each entry is hex-numbered (Boot0000, Boot0001, ...).
# Delete a specific one:
sudo efibootmgr -B -b 0007        # delete entry Boot0007

# Or, more conservatively, just deactivate it first:
sudo efibootmgr -A -b 0007        # mark Boot0007 inactive

Only delete entries you can clearly identify as stale (e.g. "ubuntu (on /dev/disk/by-partuuid/<gone>)", "USB device (vanished)"). Do not delete the live entry that loaded your current kernel — cross-reference with cat /sys/firmware/efi/efivars/BootCurrent-*.

Step 4 — cancel a wedged pending MOK request

If a previous mokutil --import or mokutil --delete was started but never confirmed in MokManager, the request sits in NVRAM consuming space and the next import sees a "full" region.

# See what is queued:
sudo mokutil --list-new
sudo mokutil --list-delete

# Cancel any pending import:
sudo mokutil --revoke-import

# Cancel any pending delete:
sudo mokutil --revoke-delete

# Sanity check: should both print "MokNew is empty" / "MokDel is empty".
sudo mokutil --list-new
sudo mokutil --list-delete

Step 5 — last resort: wipe and reseed the entire MOK list

If you have legitimately enrolled many MOKs and need to start over with a single one (the typical case for a custom-built kernel module signing setup):

# Schedule a full MOK reset on next boot. Asks for a one-time password.
sudo mokutil --reset

# After reboot, in the blue MokManager screen:
#   - choose "Reset MOK" -> "Continue"
#   - enter the password
#   - reboot
# All currently enrolled MOKs are removed. Re-enrol your one custom MOK with:
sudo mokutil --import /root/module-signing/MOK.der

Step 6 — BIOS-level last resort (ASRock Z170 PG and friends)

If steps 2–5 still leave NVRAM full, the firmware itself is holding state that Linux cannot reach. Order of operations:

  1. BIOS Setup → Security → Secure Boot → Key Management → Restore Factory Keys (or "Reset to Setup Mode" then "Install default keys"). This is the "right" reset path on ASRock 100/200-series boards.
  2. BIOS Setup → Exit → Load UEFI Defaults — the catch-all factory-reset on ASRock. Save and exit. This resets everything else too, including XMP memory profiles and fan curves — expect to redo those.
  3. Clear CMOS via jumper or coin-cell pull. For the Z170 PG: power off, unplug, hold the front-panel power button for ~30 s to drain capacitors, move the CLRCMOS1 jumper or pull the coin-cell battery for 5–10 minutes, replace, power on. This wipes everything in NVRAM that survives a normal defaults-restore (anti-rollback counters, stuck Secure Boot vendor entries, fTPM state).
  4. BIOS update / re-flash. ASRock's Instant Flash, applied with the "Clear NVRAM" option (if your BIOS version exposes it), forces a full NVRAM re-init. Use the same BIOS version you are already on; you only want the NVRAM reset, not a firmware change.
After any of step 6, Secure Boot is back in Setup Mode (no MOK list, no db entries beyond the firmware defaults). Re-enrol your custom MOK with mokutil --import and complete the MokManager flow exactly once. Resist the urge to re-enrol "just in case" — every duplicate enrolment consumes more of the limited NVRAM.
Removing intel-microcode (section 8) does NOT free any MOK / NVRAM space. Microcode lives in /boot on the filesystem, not in EFI variables. If you tried that as a workaround for "MOK storage full," it was a coincidence, not a fix — the two issues are unrelated.

11. Reinstall CUDA / cuDNN (optional)

Only after the purge in step 6 has succeeded. Install the CUDA toolkit version you actually want from NVIDIA's download page — pick the Linux / Ubuntu 24.04 / deb (network or local) variant and follow the exact commands NVIDIA shows for that version. They change per release, so do not hard-code them in a script.

If you have several CUDA versions installed side by side, switch between them by repointing the /usr/local/cuda symlink:

sudo rm -f /usr/local/cuda
sudo ln -s /usr/local/cuda-13.1 /usr/local/cuda    # adjust version

Put the environment variables in a system-wide, idempotent profile snippet so you do not keep appending to ~/.bashrc every time you reinstall:

sudo tee /etc/profile.d/cuda.sh >/dev/null <<'EOF'
if [ -d /usr/local/cuda/bin ]; then
    case ":$PATH:" in
        *:/usr/local/cuda/bin:*) ;;
        *) export PATH="/usr/local/cuda/bin:$PATH" ;;
    esac
fi
if [ -d /usr/local/cuda/lib64 ]; then
    case ":${LD_LIBRARY_PATH:-}:" in
        *:/usr/local/cuda/lib64:*) ;;
        *) export LD_LIBRARY_PATH="/usr/local/cuda/lib64:${LD_LIBRARY_PATH:-}" ;;
    esac
fi
EOF
sudo chmod 0644 /etc/profile.d/cuda.sh

Then install cuDNN that matches your CUDA major version. For cuDNN 9.x on Ubuntu 24.04 (adjust the version in the filename to the latest at developer.nvidia.com/cudnn-downloads):

# Replace 9.5.1 with whatever the current cuDNN download page lists.
CUDNN_VER=9.5.1
wget "https://developer.download.nvidia.com/compute/cudnn/${CUDNN_VER}/local_installers/cudnn-local-repo-ubuntu2404-${CUDNN_VER}_1.0-1_amd64.deb"
sudo dpkg -i "cudnn-local-repo-ubuntu2404-${CUDNN_VER}_1.0-1_amd64.deb"
sudo cp /var/cudnn-local-repo-ubuntu2404-${CUDNN_VER}/cudnn-*-keyring.gpg /usr/share/keyrings/
sudo apt update
sudo apt install -y cudnn
Always confirm the exact filename on the cuDNN download page before copying that wget URL — NVIDIA rotates filenames per release. If the URL 404s, you have the wrong version string.

12. Fix a broken nvidia-settings GUI (optional)

Some Ubuntu releases shipped an nvidia-settings package (the 510 / 590 series) that crashes with double free or corruption (!prev) the moment you click Save Current Configuration. The fix is to replace it with the binary and GTK panel libraries from the upstream .run installer.

This repo ships fix_nvidia_gui_v2.sh, which does this properly with dpkg-divert (so apt cannot silently undo it) and apt-mark hold (so package upgrades will not put it back). It also has a working revert path.

sudo ./fix_nvidia_gui_v2.sh install   # auto-detects .run in current dir
sudo ./fix_nvidia_gui_v2.sh status    # show what is currently diverted
sudo ./fix_nvidia_gui_v2.sh revert    # restore the apt-shipped files
Do not run apt --reinstall install nvidia-settings after using this script — that is exactly what the diversion is protecting against. If you need to update nvidia-settings, run revert first, do your update, then run install again with the matching .run file.

13. Final cleanup and sanity check

sudo apt update
sudo apt full-upgrade -y
sudo apt autoremove --purge -y
sudo apt --fix-broken install -y
sudo ldconfig
sudo reboot

After the final reboot, run:

lsb_release -a                       # should say 24.04 / noble
ip -brief addr                       # network up?
systemctl status NetworkManager      # or systemd-networkd
nvidia-smi                           # only if you reinstalled the driver
nvcc --version                       # only if you reinstalled the toolkit

If all four return what you expect, the recovery is done.
